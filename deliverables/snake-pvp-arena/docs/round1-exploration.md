# 第 1 轮探索

## 现有代码评估

**技术栈：** React + Vite + Tailwind (client) / Express + Socket.IO + JWT + bcrypt (server) / JSON 文件存储

**已实现功能：** 注册/登录、JWT 认证、房间创建/加入、快速匹配队列、服务端权威 tick（150ms）、Canvas 渲染（含粒子特效）、基础积分/胜负统计、排行榜、移动端触控

**发现的问题和改进空间：**

1. **无倒计时** — 第二人加入后游戏立即开始，玩家毫无准备
2. **蛇眼方向固定** — 眼睛始终朝上，不随蛇头方向旋转
3. **无再来一局** — 对局结束只能返回大厅，断裂了游戏心流
4. **Canvas 固定尺寸** — 24×24 网格 × 20px = 480px，大屏浪费空间，小屏可能溢出
5. **`watchRoom` 名不副实** — 实际是玩家重连，不支持真正的观战
6. **房间不清理** — `finished` 状态的房间永远留在内存中
7. **JSON 文件存储不安全** — 并发写入可能丢数据，match 上限硬编码 50 条
8. **无音效** — 静默的游戏体验缺乏反馈感
9. **匹配算法原始** — FIFO 队列，无技能评级
10. **游戏速度恒定** — 150ms tick 从头到尾不变，缺乏紧迫感

---

## 保守方案

- **approach:** 在现有架构上做针对性 bug 修复和 UX 打磨。具体包括：
  1. 加入 3 秒倒计时（服务端 `countdown` 状态 + 客户端倒计时 overlay）
  2. 蛇眼随方向旋转（Canvas 渲染时根据 `direction` 旋转眼睛位置）
  3. 对局结束后加「再来一局」按钮（复用 `room:create` + 向对手发邀请事件）
  4. Canvas 自适应视口（计算 `CELL_PX = min(viewportW / gridW, viewportH / gridH)`）
  5. 已结束房间 30 秒后自动清理（`setTimeout` 调 `destroyRoom`）
  6. 添加 Web Audio API 简单音效（吃食物 beep、碰撞爆炸音、倒计时滴答）
  7. 游戏速度渐进加快（每吃 5 个食物 tick 间隔减少 10ms，下限 80ms）

- **why it works:** 风险极低，不改动架构，每项修改都是独立的。倒计时 + 再来一局 + 音效是用户体验的"低垂果实"，能显著提升游戏感受。

- **downside:** 不会产生差异化特色，竞品已有的标配功能补齐而已。JSON 文件存储和内存数据的根本问题没解决。

---

## 激进方案

- **approach:** 引入竞技级特性，全面提升为"可公开运营"的水平：
  1. **SQLite 替换 JSON 文件存储** — 用 `better-sqlite3`，支持并发读写、无数据上限、WAL 模式
  2. **ELO 评分系统** — 每个用户增加 `elo` 字段（初始 1000），对局结束按标准 ELO 公式更新，匹配时优先匹配 ELO 差距 < 200 的对手
  3. **真正的观战模式** — 大厅显示"进行中的对局"列表，任何登录用户可加入观看，房间内广播观众人数
  4. **对局回放系统** — 服务端记录每 tick 的所有玩家输入（direction changes），存为紧凑的 JSON，客户端可回放任意历史对局
  5. **道具系统** — 地图随机生成特殊食物：加速（2 秒内 tick 减半）、穿墙（5 秒可穿越边界）、缩短对手（对手蛇身减 3 格）
  6. **渐进难度** — 游戏随时间推移加速（tick 从 200ms 递减到 80ms），增加紧迫感

- **why it could outperform:** ELO + 观战 + 回放 = 完整的竞技生态。道具系统打破"纯操作"的单调性，让弱玩家也有翻盘机会。SQLite 解决了数据可靠性的根本问题。

- **downside:** 工作量巨大（预估 4-6 小时），道具系统需要大量平衡测试。如果 4 轮内完不成，会出现半成品。需要严格取舍优先级。

---

## 奇特但可行方案

- **approach:** "战争迷雾 + 淘汰赛"模式 — 颠覆传统贪吃蛇的全局视野假设：
  1. **战争迷雾（Fog of War）模式** — 每个玩家只能看到蛇头周围 5×5 格的区域（服务端按玩家裁剪 snapshot 再发送），地图其余部分渲染为迷雾。食物只有进入视野才可见。这将贪吃蛇从"反应力竞技"变成"探索 + 策略"博弈。
  2. **锦标赛/淘汰模式** — 4 或 8 人报名，系统自动生成对阵表。胜者晋级，败者变为观众。决赛时所有参与者观战。排行榜追踪锦标赛冠军次数。
  3. **地图变体** — 不止空白网格：加入障碍墙、传送门（蛇头碰到 A 点传送到 B 点）、缩圈机制（地图边界随时间向中心收缩，接触缩圈边界即死）。

- **why it might unlock a better outcome:** 战争迷雾是几乎所有贪吃蛇 PvP 都没有的创新特性，能立刻形成差异化。缩圈机制借鉴 battle royale 概念，自然制造紧张冲突。锦标赛模式给核心玩家持续目标。

- **downside:** 战争迷雾需要服务端为每个玩家单独裁剪和发送 snapshot（`socket.emit` 而非 `io.to(room).emit`），改动 `advanceRoom` 的广播逻辑。锦标赛需要新的大厅 UI 和对阵管理逻辑。如果只做迷雾不做其他，可能显得"花哨但不完整"。

---

## 需要挑战的假设

1. **"JSON 文件存储够用了"** — 当前 `persist()` 每次写全量数据，match 硬上限 50 条。如果做回放或锦标赛，数据量会快速增长。是时候决定是否切换到 SQLite。
2. **"150ms 固定 tick 是合理的"** — 这让游戏前期太快（新手不适应）、后期太慢（高手觉得无聊）。动态 tick 或分段速度更合理。
3. **"全局视野是理所当然的"** — 这是传统单机贪吃蛇的遗留假设。PvP 场景下，信息不对称可以创造更丰富的策略维度。
4. **"匹配就是排队"** — FIFO 队列让新手和高手直接碰面，劝退率高。即使不做完整 ELO，至少应该按胜场数粗略分段。
5. **"对局结束 = 回到大厅"** — 这是最打断心流的设计。竞品（如 slither.io）都支持即时重开。

---

## 推荐方向

**建议以保守方案为基础，精选激进方案中的 2-3 项高杠杆特性。** 具体推荐：

1. **必做（保守方案全部）：** 倒计时、蛇眼旋转、再来一局、Canvas 自适应、房间清理、音效、渐进加速 — 这是从"能用"到"好用"的基础线
2. **高优先级（激进方案精选）：** 
   - **观战模式** — 实现成本中等，但给项目加了"社交层"，评审效果好
   - **渐进难度 + 速度可视化** — 改动量小但体验提升明显
3. **如果还有余力：** 战争迷雾模式作为"特色玩法"，即使只做基础版（固定 5×5 视野）也足以成为亮点
4. **不建议本轮做：** SQLite 迁移（收益主要在运维层，对功能评审无直接影响）、完整 ELO（需要大量对局数据才有意义）、锦标赛模式（UI 工作量太大）

**核心原则：先把基础体验做到无瑕疵，再用一个差异化特性（观战或迷雾）拉开和"普通实现"的距离。**
