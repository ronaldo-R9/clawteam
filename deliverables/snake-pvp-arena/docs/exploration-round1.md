# 第 1 轮探索

> 基于对现有代码库的完整审查（server/src 已有完整后端：Express + Socket.IO + JWT + 文件 JSON DB；client/ 缺少 src/ 目录，前端未实现）。

---

## 保守方案：修补现有架构，React SPA + 现有后端

### 技术栈
- **前端**: React 18 + Vite + TypeScript + TailwindCSS
- **后端**: 沿用现有 Express + Socket.IO（已完成）
- **数据库**: 沿用 FileDatabase（JSON 文件）
- **通信**: Socket.IO（已实现）

### 架构概览
- 补齐缺失的 `client/src/` 目录，使用 React Router 实现 `/login`、`/register`、`/game`、`/stats` 四个页面
- 用 `<canvas>` 绘制游戏画面，Socket.IO 接收 `room:update` 驱动重绘
- AuthContext 管理 JWT，axios interceptor 自动附 token
- 排行榜页面 GET `/api/stats/me` + 新增 `/api/leaderboard` 端点

### 对战机制
- 房间码模式（已实现 `room:create` / `room:join`）
- 服务端权威 tick（150ms，已实现）
- 断线 → 对手判胜（已实现 `handleDisconnect`）

### 突破点
- **零重写成本**：后端不动一行，只补前端
- 可在 2 小时内交付可用产品

### 风险评估
| 风险 | 严重度 | 应对 |
|------|--------|------|
| FileDatabase 并发写入竞态 | 中 | 单进程部署 + atomic rename（已有） |
| 无自动匹配，用户体验依赖手动分享房间码 | 低 | 后续迭代加 matchmaking |
| Canvas 游戏渲染性能在低端设备可能卡顿 | 低 | 24×24 网格极小，不成问题 |

### why it works
最小工作量路径。后端已经完成 80% 的功能，只需要补齐前端 UI 层。所有核心逻辑（认证、房间、游戏循环、对战记录）已经过验证。

### downside
- 没有自动匹配系统，用户必须手动分享房间码
- 无观战模式（`room:watch` 仅限已在房间的玩家）
- FileDatabase 无法水平扩展
- 视觉效果平庸，不会给人"哇"的感觉

## 激进方案：全功能竞技平台 — 匹配队列 + ELO + 观战 + Canvas 动效

### 技术栈
- **前端**: React 18 + Vite + TypeScript + Framer Motion + Canvas 2D（带粒子特效）
- **后端**: 沿用 Express + Socket.IO，**新增** Redis（匹配队列 + 实时排行榜缓存）
- **数据库**: SQLite（通过 better-sqlite3，替换 FileDatabase）
- **通信**: Socket.IO + 新增 `matchmaking:*` 事件命名空间

### 架构概览
```
┌─────────────┐    WebSocket     ┌──────────────────┐
│  React SPA  │◄────────────────►│  Express Server   │
│  Canvas 渲染 │                  │  ├─ AuthModule     │
│  动效系统    │                  │  ├─ MatchmakingQ   │
│  音效引擎    │                  │  ├─ GameEngine     │
└─────────────┘                  │  ├─ SpectatorHub   │
                                 │  └─ StatsService   │
                                 └────────┬───────────┘
                                          │
                                 ┌────────▼───────────┐
                                 │  SQLite + Redis     │
                                 └────────────────────┘
```
- **MatchmakingQueue**: 按 ELO ±200 范围匹配，超时扩大范围，30 秒内无匹配则 AI 填充
- **SpectatorHub**: 任何已登录用户可通过房间码观看正在进行的对局，独立 Socket.IO namespace
- **StatsService**: ELO 计算（K=32 标准公式）、赛季排行榜、个人战绩图表

### 对战机制
- **匹配**: 点击"快速匹配"→ 进入队列 → 服务端按 ELO 配对 → 自动创建房间并推送双方
- **同步**: 服务端权威 + 客户端预测（输入立即本地移动，服务端 tick 校正）
- **断线处理**: 10 秒重连窗口（`room:watch` 改为重连入口），超时判负
- **反作弊**: 服务端全权处理游戏逻辑（已实现），客户端仅负责渲染

### 突破点
- **自动匹配 + ELO** 是从"玩具"到"可玩产品"的关键跃迁
- **观战系统** 增加社交维度
- **客户端预测** 显著改善手感（消除 150ms 输入延迟感）
- **Canvas 粒子特效**（吃食物爆发、死亡爆炸）大幅提升视觉冲击力

### 风险评估
| 风险 | 严重度 | 应对 |
|------|--------|------|
| 引入 Redis 增加部署复杂度 | 中 | 可用内存 Map 先行，Redis 可选 |
| ELO 系统冷启动期匹配质量差 | 低 | 初始 ELO 统一 1000，早期放宽匹配范围 |
| 客户端预测与服务端校正的视觉抖动 | 中 | 蛇的 tick 间隔 150ms，预测误差最多 1 格，插值可掩盖 |
| 工作量比保守方案大 3-4 倍 | 高 | 优先级排序：匹配 > 观战 > ELO > 特效，可分批交付 |

### why it could outperform
用户不需要在微信里交换房间码。点击按钮就能玩。ELO 给了用户回来的理由。观战让等待不无聊。这些不是锦上添花，是**留存率**的核心驱动力。

### downside
- 开发工作量大幅增加，4 轮迭代可能无法完全交付
- SQLite 迁移需要重写 storage 层
- Redis 依赖在某些部署环境不方便

## 奇特但可行方案：单文件全栈 + ASCII 像素风 + "蛇战棋"回合制变体

### 技术栈
- **前端**: 纯 HTML + CSS Grid + 零框架（< 200KB 总包体）
- **后端**: 沿用 Express，用 `express.static` 直接 serve 前端（消灭 Vite dev server）
- **数据库**: 沿用 FileDatabase
- **通信**: Socket.IO
- **渲染**: CSS Grid + emoji/unicode 字符做"像素"（🟩🔴🍎⬛），不用 Canvas

### 架构概览
```
单进程 Express
├── /api/auth/*          (已有)
├── /api/stats/*         (已有)
├── /                    → 静态 HTML/CSS/JS（内联，单文件）
└── Socket.IO            (已有)

前端：一个 HTML 文件 + 一个 <style> + 一个 <script type="module">
     CSS Grid 24×24 渲染
     emoji 做蛇身/食物/墙壁
```

**核心变体——"蛇战棋"模式**：
- 双方同时下达**3 步指令**（如 →→↓），然后同时执行
- 每回合有 5 秒决策时间
- 增加**道具格**：加速（连续 2 步）、隐身（1 回合对手看不到你）、炸弹（摧毁 3 格范围的蛇身）
- 保留经典实时模式作为备选

### 对战机制
- **匹配**: 房间码（保持简单）
- **同步**: 回合制天然解决网络延迟问题——双方提交指令后服务端统一执行并广播结果
- **断线处理**: 超时未提交指令 → 蛇自动沿当前方向前进
- **策略深度**: 3 步预判 + 道具使用时机 = 类似"同时行动"桌游

### 突破点
- **回合制消灭延迟问题**：不需要客户端预测，不需要快速 tick 同步，手机上 3G 网络也能玩
- **零依赖前端**：没有 React、没有 Vite、没有 node_modules，一个 HTML 文件搞定，加载 < 100ms
- **Emoji 渲染意外好看**：CSS Grid + emoji 在现代浏览器上渲染清晰且自带高对比度
- **策略 > 反应速度**：吸引不同类型的玩家，不只是手速党

### 风险评估
| 风险 | 严重度 | 应对 |
|------|--------|------|
| 回合制可能让"传统贪吃蛇玩家"失望 | 高 | 保留经典实时模式作为默认，回合制作为"策略模式" |
| Emoji 在不同 OS/浏览器渲染不一致 | 中 | 用 CSS 方块 + 颜色代替 emoji 作 fallback |
| 单文件架构超过 3000 行时难以维护 | 中 | 拆为 3 个文件（HTML / game.js / style.css）仍然极简 |
| 道具系统增加游戏平衡复杂度 | 中 | 先上 1 个道具（加速），验证机制后再加 |

### why it might unlock a better outcome
现有方向是"标准贪吃蛇 PvP"——这个品类已经有无数实现。回合制变体创造了**差异化**。而零框架的极简前端意味着**开发速度极快**、**部署极简**（单个 Node 进程即可），且在任何设备上都能流畅运行。

### downside
- 游戏设计风险大：回合制贪吃蛇未经市场验证
- 可能超出"贪吃蛇 PvP"的需求范围
- 单文件架构在团队协作中不友好

## 需要挑战的假设

1. **"必须用 React"** — 24×24 的网格游戏，前端复杂度极低。React 的虚拟 DOM 对这种高频更新的游戏反而是负担。纯 Canvas 或甚至 CSS Grid 都是更高效的选择。
2. **"FileDatabase 够用"** — 对于 demo 确实够用，但 `persist()` 每次写全量 JSON，50 条 match 记录硬编码上限。如果目标是"可部署的全栈应用"，至少应该用 SQLite。
3. **"实时 PvP 必须低延迟"** — 贪吃蛇的 tick 是 150ms，远大于 FPS 游戏的要求。在这个节奏下，Socket.IO 的额外开销完全可以接受，不需要 WebRTC。
4. **"房间码是唯一的匹配方式"** — 这对两个认识的人可以，但对陌生人完全不可用。没有自动匹配 = 没有陌生人对战 = 用户留存极低。
5. **"对战就是两条蛇在空白地图跑"** — 当前设计中地图没有障碍物、没有道具、没有变化。这意味着高水平对局会变成无聊的绕圈/追尾。地图元素或道具是保持趣味性的关键。

---

## 推荐方向

**核心推荐：以保守方案为基础，选择性吸收激进方案的 2-3 个关键特性。**

具体来说：
1. **基础架构**: 用保守方案的路径——补齐 React 前端，不动后端。这是最快到达"可运行"状态的路径。
2. **必须加入**: 自动匹配队列（不需要 Redis，内存 Map 即可）。这是把"demo"变成"产品"的最关键一步。
3. **强烈建议**: Canvas 渲染 + 简单粒子特效（吃食物时）。视觉冲击力提升巨大，代码量增加很少。
4. **可选加入**: 简单的地图障碍物（每局随机生成几个不可穿越的方块）。极低成本带来的对战深度提升。
5. **暂不建议**: ELO、观战、Redis、回合制变体。这些在 4 轮迭代内可能无法充分实现和测试。

**一句话**：先跑起来，再跑得好看，最后跑得有深度。不要在架构上冒险，在游戏体验上冒险。
